### ***HTTP와  그 특징***
```
1. 텍스트 기반의 통신 규약으로 인터넷에서 데이터를 주고받을 수 있는 프로토콜 이다.
2. HTTP 동작 : 클라이언트가 어떠한 서비스를 url을 통해 request를 하면 서버에서는 해당 요청사항에 맞는 결과를 찾아서 클라이언트에게 response를 하는 형태로 동작한다.
3. HTTP 특징
  1. HTTP 메시지는 HTTP 서버와 HTTP 클라이언트에 의해 해석이 된다.
  2. TCP/IP를 이용하는 응용 프로토콜이다.
  3. HTTP는 원래 상태를 유지하지 않는(무상태), 비연결성 프로토콜이다.
    1. 무상태 ( Stateless )
      1. HTTP에서 서버가 클라이언트의 상태를 보존하지 않는 무상태 프로토콜이다.
      2. 무상태를 사용하는 이유 : 서버를 여러개 두는 것이 가능하다. 서버 확장에 용이
    2. 비연결성 ( Connectionless )
      1. 비 연결성을 가지는 HTTP에서는 실제로 요청을 주고 받을 때만 연결을 유지하고 응답을 주고 받고 나면 TCP/IP 연결을 끊는다.
      2. 이를 통해 최소한의 자원으로 서버 유지를 가능하게 한다.
      3. 비 연결성은 TCP/IP 연결을 새로 맺어야 함 (3 way handshake 시간 추가)
      
4. Request 요청
  1. Request method
    1. GET : 특정 리소스를 받기 위한 요청
    2. POST : 리소스를 생성하거나 컨트롤러를 실행하는 데 사용한다.
    3. PUT : 변경 가능한 리소스를 업데이트하는데 사용되며 항상 리소스 식별 정보를 포함해야 한다. ( 없으면 새로 생성, 있으면 완전 대체 )
    4. PATCH : 변경 가능한 리소스의 부분 업데이트에 사용되며 항상 리소스 식별 정보를 포함해야 한다. (일부 필드만 변경할 수 있음 )
    5. DELETE : 특정 리소스를 제거하는데 사용한다.
    
5. Response 응답
  1. Status Code( 상태 코드 )
    1. 1XX ( 조건부 응답 ) : 요청을 받았으며 작업을 계속한다.
    2. 2XX ( 성공 ) : 클라이언트가 요청한 동작을 수신하여 이해했고 승낙했으며 성공적으로 처리했음을 가리킨다.
    3. 3XX ( 리다이렉션 완료 ) : 클라이언트는 요청을 마치기 위해 추가 동작을 취해야 한다.
    4. 4XX ( 요청 오류 ) : 클라이언트에 오류가 있음을 나타낸다.
    5. 5XX ( 서버 오류 ) : 서버가 유효한 요청을 명백하게 수행하지 못했음을 나타낸다.
```

### ***데이터 베이스 용어 ( 테이블, 행 , 열, 도메인 ,스키마 )***
```
1. 테이블(Table)
  1. 행과 열로 이루어진 데이터의 집합을 테이블이라고 한다.
  2. 흔히 생각하는 엑셀을 떠올리면 편하다.
  3. 일반적인 데이터베이스에서는 행과 열만 있으면 테이블이라고 하지만, 관계형 데이터베이스 에서는 여기에 특별한 제약을 추가해서 릴레이션(Relation)이라고 부른다.
  4. 아래 조건을 충족하는 테이블만이 릴레이션이 될 수 있기 때문에 모든 릴레이션은 테이블이지만, 모든 테이블이 릴레이션인건 아니다.
  5. 모든 값은 유일한 값을 가진다.
  6. 하나의 릴레이션에서 중복되는 행이 존재하면 안된다.
  
2. 행(Row)
  1. 테이블을 구성하는 데이터들 중 가로로 묶은 데이터셋을 의미한다.
  2. 일반적으로 행은 한 객체에 대한 정보를 가지고 있다.
  3. 이또한 관계형 데이터베이스에서는 튜플 또는 레코드라는 이름으로 불린다.
  
3. 열(Colum)
  1. 테이블을 구성하는 데이터들 중 세로로 묶은 데이터셋을 의미한다.
  2. 일반적으로 열은 그 테이블의 속성을 의미하며 열을 구성하는 값들은 같은 도메인(Domain)으로 되어 있다.
  3. 이또한 관계형 데이터베이스에서는 속성(Attribute)이라는 이름으로 불린다.
  
4. 도메인(Domain)
  1. 데이터베이스에서 필드(Field)에 채워질 수 있는 값의 집합이다.
  2. 예를 들어, 도메인이 1에서 10사이의 정수인 속성의 필드에 11이나 -1처럼 도메인을 벗어나는 값 또는 "고양이"처럼 아예 자료형이 다른 값이 들어갈 수 없다.

5. 스키마(Schema)
  1. 데이터베이스의 구조를 전반적으로 기술한 것을 말한다.
  2. 구체적으로 데이터베이스를 구성하는 데이터 레코드의 크기, 키의 정의, 레코드 간의 관계 등을 정의한 것을 말한다.
  3. 사용자의 관점에 따라 외부 스키마, 개념 스키마, 내부 스키마로 구분한다.
  4. DBMS는 외부 스키마에 명세된 사용자의 요구를 개념 스키마 형태로 변환하고, 이를 다시 내부 스키마 형태로 변환한다.
  5. 외부 스키마
    1. 사용자의 입장에서 정의한 데이터베이스의 논리적 구조
    2. 데이터들을 어떤 형식, 구조, 화면을 통해 사용자에게 보여줄 것인가에 대한 명세를 말하며 하나의 데이터베이스에는 여러 개의 외부 스키마가 있을 수 있다.
    3. 일반 사용자에게는 질의어를 이용해 DB를 쉽게 사용할 수 있도록 하고 응용 프로그래머는 언어를 사용해서 DB에 접근하도록 한다.
    
  6. 개념 스키마
    1. 조직체 전체를 관장하는 입장에서 DB를 정의한 스키마.
    2. DB에 대한 모든 논리적 구조를 기술하기 때문에 데이터베이스에 하나만 존재하며, 통상 스키마라고 하면 개념 스키마를 일컫는다
    
  7. 내부 스키마
    1. 데이터베이스가 어떻게 저장 장치에 저장될 지에 대한 명세.
    2. 물리적인 저장 장치와 데이터베이스 간의 관계를 정의하므로 시스템 프로그래머나 시스템 설계자가 보는 관점의 스키마이다.
```
### ***트랜잭션 (Transaction)***
```
1. 데이터베이스의 상태를 변화시키기 위해 수행하는 작업 단위
2. 상태를 변화시킨다는 것은 SQL 질의어를 통해 DB에 접근하는 것이다.
3. 작업 단위 -> 많은 SQL 명령문들을 사람이 정하는 기준에 따라 정하는 것
4. Ex) 게시판에서 사용자가 글을 작성하고 올리기 버튼을 누른 후 게시판에 다시 돌아오면 게시판에 내가 쓴 글이 업데이트된 상태를 볼 수 있다.
5. 이때 DB 작업
  1. 올리기 버튼을 누른다. : INSERT문을 사용해 사용자가 입력한 게시글 데이터를 옮김.
  2. 게시판 새로 구성 : SELECT문을 사용해 최신 정보를 유지.
  
6. 현재 작업 단위 : INSERT문 + SELECT문
  1. 이를 통틀어 하나의 트랜잭션이라고 한다.
  
7. 즉, 하나의 트랜잭션 설계를 잘 만드는 것이 데이터를 다룰 때 많은 이점을 가져다준다.
8. `SELECT
   INSERT
   DELETE
   UPDATE`


9. 트랜잭션 특징
  1. 원자성(Atomicity) : 트랜잭션이 DB에 모두 반영되거나 혹은 전혀 반영되지 않아야 한다.
  2. 일관성(Consistency) : 트랜잭션의 작업 처리 결과는 항상 일관성 있어야 한다.
  3. 독립성(Isolation) : 둘 이상의 트랜잭션이 동시에 병행 실행되고 있을 때, 어떤 트랜잭션도 다른 트랜잭션 연산에 끼어들 수 없다.
  4. 지속성(Durability) : 트랜잭션이 성공적으로 완료되었으면, 결과는 영구적으로 반영되어야 한다.


10. Commit
  1. 하나의 트랜잭션이 성공적으로 끝났고, DB가 일관성있는 상태일 때 이를 알려주기 위해 사용하는 연산이다.
  
11. Rollback
  1. 하나의 트랜잭션 처리가 비정상적으로 종료되어 트랜잭션의 원자성이 깨진 경우
  2. 트랜잭션이 정상적으로 종료되지 않았을 때, last consistent state(ex. 트랜잭션의 시작 상태)로 roll back할 수 있다.
```
